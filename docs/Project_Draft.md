# CCBDA Project First Draft


## 1. Introduction

The process of applying for jobs is often repetitive, time-consuming, and inefficient. Candidates are expected to tailor their CV and cover letter for each position, research the company, and prepare for interviews, yet many end up reusing generic documents. This reduces the quality and relevance of their applications, lowering their chances of success.

This project proposes the development of a cloud-based Job Application Assistant that streamlines the process. Users upload their CV once and provide job descriptions for positions of interest. The system then leverages document processing and AI to automatically generate personalized outputs, including skills-gap analyses, tailored CVs, and cover letters.

The application is built as a complete end-to-end cloud solution on AWS. It integrates services such as Amazon Textract for CV parsing, Amazon Bedrock for LLM-powered analysis and generation, FastAPI as backend, and a modern frontend, all deployed using scalable and decoupled cloud architecture.

The primary goal of the project is twofold: to create a practically useful tool for job seekers and to demonstrate the design and implementation of a realistic, production-like cloud application using modern AWS services and best practices.


## 2. Functionality

The application is delivered as a web interface built with React/Next.js on the frontend and a FastAPI backend, with user authentication handled either through a basic session mechanism or AWS Cognito. The scope is divided into core features, which form the minimum viable product, and optional extensions that we will pursue if time permits. The subsections below follow the typical user workflow: building a profile, submitting a job description, receiving AI-generated outputs, and tracking applications over time.

### User Profile and CV Parsing

The starting point for every user is the creation of their candidate profile. The user uploads their CV as a PDF, and the upload triggers an asynchronous processing pipeline (S3 event → Lambda → Textract → Bedrock → DynamoDB), where Textract extracts the raw text from the PDF and the LLM interprets it into a structured profile (skills, experience, education, etc.). The frontend uses short polling against the API to detect when the parsed result is ready, keeping the UI stateless and avoiding persistent connections such as WebSockets. Because a CV alone rarely captures everything relevant to a job application, the system follows up with a short questionnaire containing predefined questions (e.g. preferred industry, willingness to relocate) and, optionally, additional questions generated by the LLM based on gaps detected in the parsed CV. The resulting profile serves as the foundation for all subsequent analysis steps and generations. Users can revisit and update their profile at any time to reflect new experience or preferences.

### Job Description Input

Once a profile exists, the user submits job descriptions they want to apply to. A job description can be provided by pasting text directly into the interface or by uploading a document. The system extracts and stores the key requirements, responsibilities, and qualifications from the provided text so they can be compared against the user's profile. Each submitted job description becomes a separate record tied to the user's account.

### Skills-Gap Analysis

For each job description, the system performs an automated comparison between the candidate profile and the job requirements. This analysis is carried out by the LLM via Amazon Bedrock and produces:

- A match score indicating overall alignment between the candidate and the role.
- A breakdown of matched, partially matched, and missing skills or qualifications.

### Tailored CV Generation

Based on the skills-gap analysis, the system generates the content for a tailored version of the user's CV that is reworded to better match the target position. The output focuses on textual content only, including reordered sections, adjusted emphasis, and surfaced relevant experience, rather than attempting to replicate the visual formatting of the original CV. This avoids the complexity of matching custom layouts and lets the user apply the suggested content to their own template.

### Cover Letter Generation

In addition to the tailored CV, the system generates the content for a cover letter specifically written for the position. The letter draws on the candidate profile and the skills-gap results to highlight relevant strengths and proactively address gaps. As with the CV, the output is plain text content that the user can paste into their own formatting.

### Application Dashboard

All submitted job descriptions and their associated outputs are accessible through a simple dashboard. For each application, the user can view the skills-gap analysis, the generated CV content, and the cover letter. The initial version is kept simple, serving as a single point of reference for tracking ongoing applications and revisiting previously generated material.

### Optional Features

The following extensions are not part of the MVP but will be pursued if the core system is completed ahead of schedule:

- **Job description scraping:** Users provide a URL to a job posting and the system automatically extracts the job description from the page, removing the need to copy-paste text manually.
- **Enhanced dashboard:** Adding filters, application status tracking (e.g. applied, interviewing, offer), and side-by-side comparison of multiple roles.
- **Iterative refinement:** Allowing users to review generated CV content and cover letters and perform adjustments through the UI. A more advanced version could also support a conversational refinement flow.
- **Gap-closing suggestions:** Extending the skills-gap analysis with concrete recommendations on how the candidate could address identified weaknesses, for example by highlighting transferable experience or suggesting areas to develop.
- **Interview preparation:** Generating likely interview questions tailored to the specific role and company, along with suggested talking points derived from the candidate's profile and the skills-gap analysis.


## 3. Motivation

This project provides an opportunity to apply the knowledge gained throughout the course in a more practical context, rather than focusing solely on isolated lab exercises. The project involves designing and implementing a complete system in which key cloud architecture principles such as designing for failure, elasticity, decoupled components, and security play an important role. It builds on previous lab work with core AWS services including S3, Elastic Beanstalk, Lambda, and CloudWatch, while also extending the scope to additional services such as Bedrock, Textract, and Cognito. Furthermore, applying the twelve factor methodology in practice, for example through environment based configuration, stateless processes, and centralized logging, contributes to a better understanding of how modern cloud applications are structured and maintained. In this way, the project helps bridge the gap between theoretical knowledge and practical implementation.

The project was also selected due to its relevance to our group. As students preparing to enter the job market full time, the ability to create and improve a strong CV is of high importance. Focusing on CV analysis therefore provides a clear and practical use case, which increases motivation. Compared to more generic applications, this approach offers a more meaningful context while still covering a broad range of AWS services and their integration. The system includes both synchronous components, such as API requests, and asynchronous processes such as document processing pipelines, which supports a deeper understanding of event driven architectures. In addition, the integration of a large language model through Bedrock introduces an aspect of modern cloud development that goes beyond traditional CRUD based systems.


## 4. Resources and Services

The application is built with a combination of AWS managed services and open-source tools, selected to keep the components decoupled, stateless, and elastic. Several services, such as S3, Lambda, Elastic Beanstalk, and CloudWatch, build directly on what we have used in the lab sessions, while others such as Textract and Bedrock extend the project into areas not covered in the course.

**Amazon S3** serves as the object store for user-uploaded documents such as PDF CVs and job description files. Its event-notification capability triggers the processing pipeline automatically when a new file is uploaded.

**Amazon Textract** extracts raw text from uploaded PDF CVs, including tables and key-value pairs. As a fully managed service, it removes the need to maintain custom OCR infrastructure. The raw output is then passed to the LLM for interpretation into a structured profile.

**AWS Lambda** handles the event-driven asynchronous processing: it is triggered on S3 uploads, runs Textract parsing, invokes Bedrock for LLM inference, and writes the results to DynamoDB.

**Amazon SQS** decouples the asynchronous Lambda processing pipeline from the synchronous API and notifies the backend when results are ready. This ensures that processing spikes do not overload the backend and that no uploads are lost.

**Elastic Beanstalk** hosts the containerised FastAPI backend with built-in auto-scaling, load balancing, and health monitoring. This provides us with simplified deployments and was already practised in the lab sessions.

**Amazon Bedrock** provides access to large language models for interpreting parsed CV text into structured profiles, generating tailored CVs and cover letters, and performing skills-gap analysis, without requiring us to host or fine-tune a model.

**Amazon DynamoDB** stores user profiles, parsed CV data, job records, and generated outputs such as tailored CVs and cover letters.

**Amazon CloudFront** serves the static React frontend build directly from S3, caching assets at edge locations to reduce latency for end users.

**Amazon CloudWatch** provides centralised logging and monitoring across all services, supporting both debugging during development and operational monitoring in production.

**Amazon Cognito** is planned as a stretch goal to add user authentication and authorisation via managed user pools and JWT tokens. The MVP will either omit authentication entirely or use a simple approach.

Beyond AWS, the project uses Python with FastAPI as the backend REST framework and React for the frontend. Docker is used to containerise the application for reproducible Elastic Beanstalk deployments. Version control and CI/CD are handled through GitHub and GitHub Actions.


## 5. 160h Plan

The 160 hours allocated to this project are distributed across seven main activity categories and a contingency buffer. The breakdown below details the rationale for each estimate and the expected deliverables.

### Design & Research (20h)

This phase covers both the initial architectural design and the investigation of unfamiliar AWS services. On the design side, tasks include selecting the appropriate AWS services, drawing the system architecture diagram, defining REST API contracts, and specifying the DynamoDB schema. On the research side, particular attention is given to Amazon Textract and Amazon Bedrock, which are not covered in depth in the course labs. Time is allocated to study their documentation and validate assumptions through small proof-of-concept scripts before committing to a final design. This phase is front-loaded and must be completed before development begins.

### Backend Development (45h)

This is the largest single workstream and covers the full implementation of server-side logic. It includes the REST endpoint implementation (upload, analyse, retrieve results), the asynchronous document processing pipeline (S3 trigger → Textract → Bedrock → DynamoDB), prompt design and Bedrock integration for cover letter generation and skills-gap analysis, and error handling with CloudWatch logging. The pipeline is the most complex component because it spans multiple services and must handle failures gracefully, which justifies the larger allocation compared to other workstreams.

### Frontend Development (20h)

The frontend provides the user-facing interface through which candidates upload their CV, submit job descriptions, and view the generated outputs. Tasks include designing the page layout, integrating with the backend REST API, and implementing the results display covering the skills-gap summary, tailored CV content, and cover letter.

### Infrastructure & Deployment (20h)

All infrastructure is defined as code to ensure reproducibility and alignment with the Twelve-Factor methodology. This phase covers writing CloudFormation templates for all AWS resources, configuring a CloudFront distribution in front of S3 and API Gateway, and setting up a GitHub Actions pipeline for automated build, test, and deploy on every merge to `main`.

### Testing & Debugging (15h)

Quality assurance is spread across the project timeline rather than concentrated at the end. The allocated hours cover unit tests for Lambda functions and utility modules, integration tests verifying the end-to-end pipeline, and manual testing with realistic CVs and job descriptions.

### Meetings & Coordination (10h)

We hold weekly synchronisation meetings and periodic check-ins with the course instructor. Additional time accounts for asynchronous coordination overhead such as pull request reviews and task tracking.

### Documentation & Presentation (20h)

Documentation is produced iteratively alongside development. Architectural and design decisions are captured continuously as the project evolves. The final report, incorporating results, evaluation, and lessons learned, is written at the end. Presentation slides for the final demo are prepared in parallel.

### Buffer (10h)

Reserved for unforeseen technical difficulties such as IAM permission issues, unexpected service quotas, or cross-component integration bugs. If the core system is completed ahead of schedule, this time may be redirected to optional stretch features.
